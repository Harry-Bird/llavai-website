# Llavai Launch Audit — 2026-06-14

Verified against live Supabase (`bwgaolpmarlnwbcrklzc`), the live n8n estate, live Stripe, and
deployed www.llavai.com. Six-dimension multi-agent audit (frontend, Supabase, n8n, integrations,
notifications, scalability) with adversarial verification of every critical/high/security finding.
53 findings; 17 confirmed critical/high. Two initial findings were **refuted/down-ranked** by the
verifier (see notes) — do not act on the un-corrected versions.

---

## 1. Verdict

A paying customer who signed up today gets almost nothing they were sold. Every load-bearing piece
of the *value* (not the plumbing) is starved or silent: the feed is empty for 100% of real users,
no one is notified when a listing lands or a viewing is booked, and the flagship Julia call chain
has never completed a single real call end-to-end. The machinery underneath is sound — tier/entitlement
logic, idempotent call-credit metering (can't double-spend or go negative), RLS, Stripe live-mode
config, and teaser redaction all verified correct. This is a **correctly-built engine with no fuel
and no dashboard lights.** Plus: the launched product **takes money on a Supabase Free plan with zero
backups** — an existential data-loss risk independent of everything else. Do not run another marketing
push until sections 3–4 are cleared.

## 2. The five owner questions

1. **Old/dead workflows still firing?** No. Every old workflow is `active:false` and none has fired
   since ~2026-06-11 14:30. The one legacy workflow still active is **Profile Creation** (`XQU6SaN7`),
   which still writes lead **contact** data (name/email/phone — *not* financials, see M-1) to a Google
   Sheet. Quota healthy: ~1,300–1,400 of 2,500 exec/mo, dominated by the 10-min drain. *Archive the ~12
   dead workflows so an accidental re-activate can't resurrect double-processing.*
2. **What still needs building?** The entire client-notification layer; consent/unsubscribe infra;
   the EU AI Act Art. 50 disclosure in Julia's live prompt; `julia-intro.mp3`; and the onboarding capture
   that makes the feed work (required `idealista_url` + forwarding verification). See §6.
3. **Do all promised features work end-to-end?** No. Working: credits/packs (W7), tier gating,
   billing/portal, documents/messages/viewings surfaces, teaser redaction. Broken/unproven: the organic
   feed (empty), Julia calls (never completed a real call), Pro monthly allowance reset (W3 writes
   `current_period_end=NULL`), client notifications (absent). Zero completed real purchases.
4. **100+ concurrent users?** Not as configured. Static frontend + PostgREST reads scale fine. Four
   walls break before 100: (1) Supabase Free (no backups, 5 GB egress); (2) feed `select('*')` ships a
   ~60 kB/row blob → egress cap at ~93 paid users; (3) Apify cost + sequential scrape loop; (4) n8n
   quota if `page_view` analytics is ever switched on.
5. **Clients notified of new listings?** No — of nothing that matters. 8 of 9 outbound email paths go
   to ops; the 9th is the forwarding-code relay. No new-listing alert, no viewing-booked confirmation,
   no in-app badge, no trigger on `listings`/`viewings`. Highest-leverage churn driver in the product.

## 3. CRITICAL — fix before taking another payment

- **C1. Supabase on FREE plan, no backups.** `get_organization` → `plan:"free"` (no PITR, 5 GB egress,
  500 MB DB, inactivity-pause). Live paying + PII data exists. One bad migration / DELETE / service-role
  slip = permanent loss. **Fix: upgrade to Supabase Pro (~$25/mo) now; nightly `pg_dump` off-site until then.**
- **C2. Feed empty for 100% of real users.** `feed_search_clients` returns 0 rows (view requires
  `profiles.idealista_url`; 0 of 6 profiles set it — it's optional, profile.html:636). Feed Backfill
  (`lc7n2PB`) iterates zero clients every 6h (exec 3217: node returns `[[]]`, ~0.9 s, green "success").
  Only 3 listings platform-wide, all owner accounts. `alert_email_verified=0` for all 6 → CloudMailin
  feed source also dead. **Fix: make `idealista_url` required (or derive a default search) at onboarding;
  backfill; surface "feed inactive" state; alert when `feed_search_clients=0` while paid subs > 0.**
- **C3. No client notified of a new listing or booked viewing.** No trigger on `listings`; W1/Backfill
  end at a no-op; W2's booking branch has no client email; no realtime/polling/badge in account.html.
  **Fix: AFTER-INSERT trigger on `listings` → pg_net → batched daily digest (never per-upsert); AFTER-UPDATE
  on `viewings`→booked → client email + `.ics` (or a `sender='team'` messages row). Gate behind consent (H4).**

## 4. HIGH — broken / unbuilt promised features

- **H1. W3 writes `current_period_end=NULL` → Pro 60-call allowance never resets.** The value moved to
  `items.data[0].current_period_end` in the 2026 Stripe API; W3 reads the old top-level path → `undefined`
  → NULL. Renewal-reset trigger fires on `IS DISTINCT FROM`, and `NULL IS DISTINCT FROM NULL = FALSE`, so
  allowance decays to 0 each cycle. Initial activation still grants 60; only renewals break. **Fix: read
  `items.data[0].current_period_end` in W3, republish, backfill rows.**
- **H2. Julia call chain armed but never completed a real call.** W1-GATED v2: 2 lifetime execs, both
  bailed before dial. `call_attempts` completed/failed = 0 (2 rows `skipped`); `viewings source='julia'`
  = 0; ledger = 0. Five stages (CloudMailin secret routing, Apify, Retell dial, post-call writeback) all
  unvalidated in production. W2 gates on a specific `agent_id` — verify the live agent is whitelisted.
  **Fix: one true end-to-end smoke test before the first paying Pro listing (in-hours + after-hours).**
- **H3. EU AI Act Art. 50 disclosure unverified in the live Retell prompt.** Statutory 2026-08-02 (49 days).
  Spec is "PROPOSAL ONLY"; the lone "cleared per owner" claim has zero corroboration; prompt unreadable here.
  **Fix: pull the published agent prompt / a transcript; confirm turn-1 disclosure; ship via draft→publish.**
- **H4. No consent / unsubscribe / notification-preference infra.** Blocks safely shipping C3 to an
  EU audience (GDPR consent + one-click List-Unsubscribe). **Fix: `marketing_consent_at` + per-channel prefs
  + signed unsubscribe tokens + account.html toggle. Transactional = no opt-in; digests = explicit opt-in.**
- **H5. Feed `select('*')` egress wall ~93 paid users.** account.html:1005 ships ~60 kB/row (87% never
  rendered). **Fix: select the ~13 columns `card()` uses + `.limit()` → ~8× egress cut; moves cap to ~700+.**
- **H6. Drain has no per-user fairness and no retry on telephony failure.** FIFO `limit=3`; a telephony
  error marks the row failed + ops-emails with **no requeue** → a paid call silently dropped. *(Verifier
  correction: the "18 calls/hr can't honour 60 s" framing is largely refuted — in-hours listings fire Retell
  directly via W1's unthrottled path; the drain only flushes the overnight backlog.)* **Fix: requeue-with-backoff;
  per-user round-robin; surface queue depth.**

## 5. MEDIUM / scale-soon / security-hardening

- **M-1. Legacy "Profile Creation" still active, writes lead contact data to a Google Sheet.**
  *Verifier-corrected: NOT a financial-PII leak* — the Set-node maps body keys the forms never send, so
  every Sheet row's financial columns are null (verified execs 3108 + 2957; profile.html only POSTs
  name/full_name/email/phone/status). Real issue = contact details duplicated to a Sheet outside RLS +
  an undisclosed processing activity. **Fix: repoint forms at an alert-only webhook (no Sheets write) or
  strip the node, then archive.** *(Do not describe to anyone as a financial leak.)*
- **Lost-lead (narrow):** for **not-signed-in** users the full profile lives only in `localStorage`
  (`llavai-profile-pending`, profile.html:1081) until the magic link is clicked and `/account` flushes it.
  Never complete auth / clear storage / different device → rich profile is lost (ops still gets the contact ping).
- **Stripe:** paid path never completed live (0 charges/active-subs/ledger rows). A `Stripe account TEST`
  credential coexists with the live one in n8n — confirm W7/W3 use LIVE, delete TEST. W4 v2 pack branch
  builds malformed params (400, exec 3109) — latent (packs sell via Payment Links/W7). **Fix: one real €12
  pack + one Pro checkout as launch gates (refund after).**
- **Apify** sequential `splitInBatches:1` (290 s/client) won't finish 100 clients in 6h; ~$6/mo/client
  (~$612/mo at 100). Parallelise, dedupe shared URLs, cap results, add spend alert.
- **Funnel Ingest = 1 n8n exec per beacon**; `page_view` is in the allow-list (never fired). Enabling it
  blows the 2,500/mo cap. **Fix: write analytics straight to `funnel_events` (RLS insert / edge fn), not n8n.**
- **Security:** `notify_new_document`/`notify_new_user_message` are anon-EXECUTABLE SECURITY DEFINER fns.
  *Verifier-corrected to LOW:* the anon ops-spam vector is **refuted** (PostgREST won't expose `RETURNS trigger`
  fns — PGRST202; null `NEW` swallowed before `net.http_post`). **Residual real issue: the n8n webhook secret
  is stored plaintext in the function body — rotate it + move it out; `REVOKE EXECUTE … FROM anon, PUBLIC` for hygiene.**
- **~22 RLS policies re-evaluate `auth.uid()` per row** (auth_rls_initplan) — wrap as `(select auth.uid())`.
- `call_queue.listing_id` FK unindexed; feed query is a Seq Scan (add user_id predicate + partial index).
- `current_tier()` maps legacy `plan='spot'` active → 'essential' (1 live row). Normalise + use an allowlist.
- `subscriptions.call_allowance` default = 300 but Pro trigger forces 60 — latent over-grant; default to 0/60.
- W5a (Pro Application Alert) published with 0 execs ever — archive or wire.

## 7. Scale ceiling (break-order)

| Users | What breaks |
|---|---|
| Any (now) | Supabase Free = no backups — data-loss risk at user #1 |
| ~10–50 | Call-queue fairness + no-retry; Retell single-number concurrency |
| ~50–100 | Apify cost + sequential scrape loop can't finish a 6h cycle |
| ~93 | Feed `select('*')` egress blows the 5 GB free cap (trim → ~700+) |
| ~100+ | n8n 2,500 exec/mo cap (instant if `page_view` enabled); RLS per-row `auth.uid()` + Seq-Scan latency |
| Never | Vercel static frontend + stateless PostgREST reads |

**Single most urgent scale fix:** upgrade Supabase off Free **and** trim the feed query — one combined
change removes the data-loss risk and ~90% of feed egress.
